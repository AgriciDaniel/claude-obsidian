> DragonScale は LLM wiki vault 向けの 4 メカニズム構成、オプトインのメモリレイヤーになりました。

![DragonScale Memory mechanism overview](https://raw.githubusercontent.com/AgriciDaniel/claude-obsidian/main/wiki/meta/dragonscale-mechanism-overview.png)

## TL;DR

- 出荷済み: DragonScale には fold ロールアップ、決定論的ページアドレス、semantic tiling lint、boundary-first autoresearch が揃いました。
- ゲート条件: `bash bin/setup-dragonscale.sh`、加えてメカニズムごとのローカル依存。
- 出荷していないもの: fold-of-folds、キャリブレート済みの tiling しきい値、ホスティングモード、マルチ vault ロールアップ。

## 4 つのメカニズム

<!-- overview diagram rendered at the top of the page. -->

### 1. Fold operator

What:
`wiki/log.md` のエントリを extractive にロールアップし、`wiki/folds/` 配下の fold ページにする。

Invoke:
`fold the log, dry-run k=3` を依頼し、出力が正しければ `fold the log, commit k=3` を依頼する。

Skip:
ログにまとまったバッチがないとき、もしくは削除ベースのコンパクションをしたいときはスキップ。
fold は要約します。
ガベージコレクションはしません。

Status:
`1.5.0` で出荷済み。

First real run:
`wiki/folds/fold-k3-from-2026-04-23-to-2026-04-24-n8.md`。

Observed output:
115 行。
8 件の子。
フラットな extractive fold。

### 2. 決定論的ページアドレス

What:
新しい非メタ wiki ページに対する安定した `address: c-NNNNNN` frontmatter。

Invoke:
`./scripts/allocate-address.sh` を直接実行するか、DragonScale セットアップ後に `wiki-ingest` にアドレスを割り当てさせる。

Skip:
安定したページ ID が不要なとき、もしくは複数の書き手が同時にアドレスを割り当てる可能性があるときはスキップ。

Status:
`1.5.0` で出荷済み。

Format:
`c-000042`。

Important limit:
これは creation-order カウンターです。
コンテンツハッシュではありません。

Validation run:
カウンターは 2 から 3 に進みました。
`c-000002` を返しました。
未割り当ての予約ギャップは仕様上許容範囲です。

### 3. Semantic tiling lint

What:
wiki に対する埋め込みベースの重複ページ検出。

Invoke:
準備状況には `python3 ./scripts/tiling-check.py --peek`、保存レポートには `python3 ./scripts/tiling-check.py --report wiki/meta/tiling-report-YYYY-MM-DD.md` を実行。

Skip:
`ollama` または `nomic-embed-text` が利用できないとき、もしくは未キャリブレートのレビューバンドが信号より雑音を生むときはスキップ。

Status:
`1.5.0` で出荷済み。

Default model:
`ollama` 経由のローカル `nomic-embed-text`。

Default bands:
`>= 0.90` を error。
`0.80` から `0.90` を review。
`0.80` 未満を pass。

Important limit:
デフォルトはシード値です。
この vault に対してキャリブレートされていません。

First full report:
`wiki/meta/tiling-report-2026-04-24.md`。

### 4. Boundary-first autoresearch

What:
wiki ページのフロンティアスコアリング。トピックなしの `/autoresearch` の候補選択に使われる。

Invoke:
DragonScale が有効な vault でトピックなしの `/autoresearch` を実行するか、`python3 ./scripts/boundary-score.py --json --top 5` を実行。

Skip:
明示的なトピック制御だけが欲しいとき、もしくは DragonScale を純粋なメモリ操作に限定したいときはスキップ。

Status:
`1.6.0` で出荷済み。

Formula:
`boundary_score(p) = (out_degree(p) - in_degree(p)) * exp(-days_since_updated / 30)`。

What it does:
外向きにリンクし、流入リンクが少なく、まだ役立つ程度に最近のページを見つける。

What it is not:
これは純粋なメモリではありません。
agenda コントロールです。
トピックは依然としてユーザーが選びます。

First run artifact:
`wiki/meta/boundary-frontier-2026-04-24.md`。

Top frontier facts:
22 件のスコア対象ページ。
ポジティブスコアは 7 ページ報告された。
最高スコアは `4.693`。
`DragonScale Memory` は `4.000` を獲得。

## ドラゴン曲線について簡単に

Heighway のドラゴン曲線は paperfolding の再帰によって構築されます。
各構築ステップは、ターン列をより大きな自己相似列にフォールドします。
ターン列は正則 paperfolding 列で、OEIS A014577 です。
[Dragon curve](https://en.wikipedia.org/wiki/Dragon_curve) を参照してください。

その境界はおよそ `1.5236` のフラクタル次元を持ちます。
この数字はここでは証明ではなく、方向感覚として有用です。
境界が単純な線でも埋まった平面でもないことを表しています。
構造化されたフロンティアです。

これがメモリレイヤーの良いアナロジーになります。
再帰構造は階層的なロールアップを示唆します。
自己相似的な境界は frontier-first な agenda を示唆します。
ただしこれはアナロジーであって、導出ではありません。
DragonScale が曲線の幾何から最適なメモリ挙動を証明するわけではありません。

## v1.6.0 に含まれないもの

- fold-of-folds なし。
- 1 つのフラットなコミット済み fold を超える階層レベルの積み重ねなし。
- 子ページの自動削除やコンパクションなし。
- キャリブレートされた M3 しきい値なし。
- この特定の vault に対する文献裏付けの M3 バンドなし。
- tiling 結果からの自動マージなし。
- M4 についての純粋メモリの主張なし。
- M4 における隠れたトピック選択なし。
- ホスティングまたはマネージドモードなし。
- マルチ vault の fold ロールアップなし。
- マルチ書き手のアドレス割り当て保証なし。
- boundary scoring における alias 対応の完全な Obsidian wikilink リゾルバーなし。
- 永続的な boundary-score キャッシュなし。
- 自動 autoresearch トリガーなし。

## v1.4.x からのアップグレード

ベースユーザーには非破壊です。

アップグレードパス:

1. `main` をプル。
2. エージェント設定が要求するならプラグインとして再インストール。
3. DragonScale が不要なら、通常の v1.4.x wiki ワークフローを使い続ける。
4. 準備ができたときだけ DragonScale にオプトイン:

```bash
bash bin/setup-dragonscale.sh
```

オプトインしなければ、v1.4.x のワークフローから何も変わりません。

DragonScale は機能検出されます。
`wiki-ingest`、`wiki-lint`、`/autoresearch` は、セットアップ状態やヘルパー依存が欠けていればフォールバックします。
