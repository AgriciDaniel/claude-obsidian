# 2026-04-24 v1.6.0リリースセッション

v1.5.0、v1.5.1、v1.6.0を扱った同日内のリリースサイクル。

このセッションでは、DragonScaleを3メカニズムのオプトイン拡張から4メカニズムのオプトイン拡張へと進め、その途中にハードニングのポイントリリースを挟んだ。

**公開リリースノート**: [docs/releases/v1.6.0.md](../../docs/releases/v1.6.0.md)。これがユーザー向けの成果物であり、`gh release create v1.6.0`本文へ貼り付ける情報源。本ページはエンジニアリング用の内部記録。

このページの真実の源は以下:

- `CHANGELOG.md`の`[1.5.0]`、`[1.5.1]`、`[1.6.0]`エントリ
- `wiki/log.md`のPhase 3.5、Phase 3.6、Phase 4を扱う2026-04-24エントリ
- ファイル存在、タグ、コミット履歴に関する現在のリポジトリ状態

## リリース順序

- `v1.5.0`: Phase 3.5、午前。フェーズ横断のハードニング、オプトインのDragonScaleインストーラー、再現可能テスト、`CHANGELOG.md`、`Makefile`、バージョン同期。
- `v1.5.1`: Phase 3.6、午後。tilingのセキュリティ、ロールアウトの基準日、AGENTSドキュメント、wiki-ingestの文言、インストールガイドのバージョンに関する5件のピンポイントなハードニング修正。
- `v1.6.0`: Phase 4、夕方。DragonScaleメカニズム4をオプトインの境界優先autoresearchとして出荷、新しいスコアラースクリプト、新しいテスト、トピック無しautoresearchの候補フローを伴う。

## リリース概要

午前のリリースv1.5.0は、Phase 1からPhase 3までで蓄積されたhold-ship監査項目を解消するためのハードニングパスだった。

午後のリリースv1.5.1は、残った5件の不整合と、Phase 4を進める前のtilingの書き込み先制限の問題を直す絞り込み修正パスだった。

夕方のリリースv1.6.0は、これまで先送りされていた4番目のDragonScaleメカニズムを出荷し、`[[DragonScale Memory]]`に記載のオプトインDragonScaleセットを完成させた。

v1.5.0、v1.5.1、v1.6.0いずれもgitタグは作成されなかった。

このリリースサイクル中に`origin/main`へのpushは行われなかった。

`git tag`を確認すると、ローカルにはDragonScale以前のタグだけが存在する:

- `v1.1`
- `v1.4.0`
- `v1.4.1`
- `v1.4.2`
- `v1.4.3`

## 2026-04-24のフェーズタイムライン

### 午前: Phase 3.5からv1.5.0へ

これはフェーズ横断のハードニングリリースだった。

ゴールは、それ以前のDragonScaleフェーズで実施した整合性監査が見つけた10件のhold-ship項目を解消することだった。

このリリースでは、欠けていたインストーラー、テストハーネス、changelogの成果物が追加され、拡張は部分的にしか文書化されていない状態から再現可能な状態になった。

メカニズム4はこの時点で意図的に未出荷のまま残された。

仕様は依然として未実装と表記し、`/autoresearch`は変更しなかった。

このリリースでは、プラグインとマーケットプレースのバージョン文字列を`1.5.0`に同期させた。

### 午後: Phase 3.6からv1.5.1へ

これは絞り込んだハードニングのポイントリリースだった。

スコープは拡張しなかった。

正確性、整合性、セキュリティの観点で重要な5件の問題を修正した:

- tilingレポートのパス制限
- ロールアウト基準日
- AGENTSのスキル表と互換性に関する文言
- wiki-ingestの文言:不変の`.raw/`と保守される`.raw/.manifest.json`の区別
- インストールガイドのバージョンずれ

結果として、メカニズム4が提案から出荷コードに移る前に、より小さく整理されたリリース状態となった。

### 夕方: Phase 4からv1.6.0へ

これは境界優先autoresearchをDragonScaleメカニズム4として出荷した機能リリースだった。

新しいスコアラーはwikilinkグラフからフロンティア候補を計算し、DragonScale有効なvaultでトピック無しの`/autoresearch`を呼び出した際にそれらを公開する。

このモードはオプトインのままであり、実装と文書全体を通じてアジェンダ制御として位置付けた。

その後、`plugin.json`と`marketplace.json`のバージョンを`1.6.0`に同期した。

## 主要な学び

これらは本セッション時点での`wiki/hot.md`のKey Lessonsセクションから引いたもの。

1. フェーズ横断の監査は不可欠。個別のフェーズレビューはフェーズ間のドリフトを見逃す。
2. オプトインの機能検出(`[ -x script ] && [ -f state ]`)は、採用者と非採用者の双方に対して既定のプラグイン挙動を維持する。
3. PostToolUseフックのマッチャーは`Write|Edit`。Bashの書き込みでは発火しない。トラックされた状態を変えるスクリプトは副作用コミットを避けるため、Bashのみで実装する必要がある。
4. シードvaultの自己整合性が重要。仕様がポストロールアウトのページにアドレスが必要と述べるなら、概念ページ自身もアドレスを持つ必要がある。
5. Codex敵対的レビューラウンドは、執筆者が完了したと感じたときではなく、指摘リストが空になったときに止まる。
